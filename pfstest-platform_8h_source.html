4 /*
5 
6 PFSTest attempts to work out-of-the-box with any ANSI C compiler. In
7 order to do so, it makes usually-reliable guesses about platform
8 alignment requirements and the sizes of primitive types.
9 
10 If you would like to override these guesses, a plaform port can
11 define:
12 
13 PFSTEST_ALIGNMENT:
14 
15  The biggest alignment required by any type on your platform.
16 
17 pfstest_uintptr_t:
18 
19  An unsigned integer type big enough to hold the value of any
20  pointer on your platform. On C99 platforms, defaults to
21  uintptr_t. On other platforms, defaults to size_t.
22 
23 pfstest_intmax_t:
24 
25  The largest signed integer type on your platform. On C99
26  platforms, defaults to intmax_t. On other platforms, defaults to
27  long.
28 
29 pfstest_uintmax_t:
30 
31  The largest unsigned integer type on your platform. On C99
32  platforms, defaults to uintmax_t. On other platforms, defaults to
33  unsigned pfstest_intmax_t.
34 
35 PFSTEST_UINTMAX_MAX:
36 
37  The maximum value of the largest unsigned integer type on your
38  platform. On C99 platforms, defaults to UINTMAX_MAX. On other
39  platforms, defaults to ULONG_MAX.
40 
41 ====
42 
43 PFSTest supports some optional features which are not standard to ANSI
44 C but which exist on some platforms and can be enabled for your
45 convenience if your platform supports them. A platform port header may
46 additionally define:
47 
48 pfstest_constructor(name):
49 
50  If your compiler provides an extension that allows registering
51  hooks to run at start-up, this should expand to the code which
52  creates such a hook. On ports where this macro is not defined,
53  register_test and register_plugin will need to be called to set up
54  the test environment before invoking pfstest.
55 
56 PFSTEST_NORETURN:
57 
58  If your platform supports an annotation for functions to indicate
59  that they should not return, define this macro to that annotation.
60 
61 ====
62 
63 PFSTest supports Harvard architectures, and on those architectures it
64 attempts to use as little RAM as possible by storing strings and other
65 constant data in program memory. The various compilers for these
66 architectures use different mechanisms and syntax to support multiple
67 memory spaces, and much of the platform-specific configuration in this
68 module is designed to provide a common API for accessing the program
69 memory space.
70 
71 In order to enable Harvard architecture support, a platform port
72 header must define:
73 
74 pfstest_pg:
75 
76  Expands to a keyword or keywords used by the compiler to specify
77  that a data object should reside in the non-volatile memory space.
78  Note that this is only valid when creating data objects. It may
79  not be correct to use this as a general storage-class specifier to
80  describe the location of the target of a pointer (for example, on
81  avr-gcc). See also: pfstest_pg_ptr
82 
83 pfstest_pg_ptr:
84 
85  Expands to a keyword or keywords used by the compiler to specify
86  that a pointer should point to the non-volatile memory space. Note
87  that this is only valid when creating pointers. It may not be
88  correct to use this as a storage-class specifier for data objects
89  themselves (for example, on avr-gcc). If left undefined, defaults
90  to pfstest_pg. See also: pfstest_pg
91 
92 pfstest_pg_str(string):
93 
94  Expands to a string constant which resides in the non-volatile
95  memory space.
96 
97 pfstest_memcpy_pg(ram, pg, size):
98 
99  A function or macro which is a variant of memcpy which copies data
100  from the nonvolatile memory space into RAM.
101 
102 pfstest_strcmp_pg(ram, pg):
103 
104  A function or macro which is a variant of strcmp which compares a
105  string in RAM to one in the non-volatile memory space.
106 
107 pfstest_strcmp_pgpg(pg1, pg2):
108 
109  A function or macro which is a variant of strcmp which compares
110  two strings that reside in the non-volatile memory space.
111 
112 pfstest_strcat_pg(ram, pg):
113 
114  A function or macro which is a variant of strcat which
115  concatenates a string which resides in the non-volatile memory
116  space onto the end of a string which resides in RAM.
117 
118  */

120 /* Platforms with built-in support are guessed using compiler-specific
121  * macro definitions. This auto-detection can be overridden in one of
122  * two ways:
123  *
124  * 1) To force one of the built-in platforms, define PFSTEST_PLATFORM
125  * to one of the macros below.
126  *
127  * 2) To use a user-defined platform, create a platform port header
128  * that meets the above requirements and define
129  * PFSTEST_PLATFORM_HEADER to a string suitable for an #include
130  * directive.
131  *
132  * Getting the quoting right when manually setting
133  * PFSTEST_PLATFORM_HEADER from the command line can be
134  * tricky. Here is the proper syntax for defining it in various
135  * environments: for double- and single-quoted CPPFLAGS under POSIX
136  * sh, and for defining a variable :
137  *
138  * On a POSIX sh make command line, value in double-quotes:
139  * make CPPFLAGS="-DPFSTEST_PLATFORM_HEADER=\\\"pfstest-platform-user.h\\\""
140  *
141  * On a POSIX sh make command line, value in single-quotes:
142  * make CPPFLAGS='-DPFSTEST_PLATFORM_HEADER=\"pfstest-platform-user.h\"'
143  *
144  * In a Makefile:
145  * CPPFLAGS := -DPFSTEST_PLATFORM_HEADER=\"pfstest-platform-gcc.h\"
146  */
